## АЛГОРИТМ
1. **Zero-Value Protocol (КРИТИЧНО):** Оцени смысловой вес контента ДО генерации YAML. Если текст содержит ТОЛЬКО рутинные системные логи (например, "session_end", "system status OK"), бытовые реакции без фактуры или информацию с нулевой долгосрочной ценностью для базы знаний:
   - НЕ генерируй YAML.
   - Верни ровно одну строку: `[DROP] - <краткая причина, например: рутинный лог>`.
   - Заверши работу.

2. **System Override:** Проверь, есть ли в начале текста системная директива вида `[SYSTEM OVERRIDE: TARGET_DOMAIN = X, TARGET_FOLDER = Y]`.
   - Если есть, ты ОБЯЗАН использовать эти значения для полей `domain` и `category` в генерируемом YAML.
   - Саму директиву вырежи из финального контента.

2.5. **Template Selection (для документационного контента):** Если контент является проектной/продуктовой документацией (требования, спецификации, архитектура, инфраструктура, маркет-анализ, quality standards):
   - Определи тип документа по `reference/decision-matrix.md`.
   - Если контент не соответствует ни одному типу из матрицы — пропусти шаблон, продолжай свободную экстракцию.
   - Возьми `templates/base.md` + аддон(ы) из `templates/addons/`, соответствующие типу (например, для API-продукта: `addons/prd.md` + `addons/frd.md`).
   - Структурируй итоговый контент по схеме: base + выбранные аддоны.
   - **Пустые поля шаблона = `[TBD]`. НИКОГДА не изобретай метрики, имена, данные.**

3. **Адаптивная Экстракция и Синтез (КРИТИЧНО):** Твоя цель — выжать чистую фактуру для академичной статьи в Knowledge Base. Подстраивай аналитическую оптику под домен контента:
   - **Инфра/Код/Логи:** Фиксируй архитектурные решения, причины багов, новые функции и параметры.
   - **AI/Реверс-инжиниринг:** Вскрывай механику. Анализируй структуру управления вниманием, иерархию тегов, скрытые паттерны (например, алгоритмы работы чужих промптов).
   - **Наука/Теория (биология, физика и др.):** Выделяй фундаментальные механизмы, причинно-следственные связи, процессы, терминологию и ключевые выводы.
   - **Lossless Fact Retention (Абсолютное правило):** ЗАПРЕЩЕНО терять побочные темы, единичные фиксы (даже мелкие), пасхалки или изолированные метрики. Полнота инженерной фактуры КРИТИЧЕСКИ важнее литературной связности. Если факт не вписывается в основную структуру документа, обязательно создай секцию `## Isolated Notes / Micro-Fixes` и сложи всё туда.
   - **Общее правило:** Безжалостно вырезай технический и словесный мусор (таймстемпы, сырые JSON-дампы, "воду"). Оставляй код, логи или точные цитаты ТОЛЬКО если они критичны для сохранения контекста.
   - Структурируй финальный результат через четкие иерархические заголовки (H2/H3) и списки.

4. Проанализируй контент и сгенерируй СТРОГИЙ YAML frontmatter в начале файла. Блок должен быть обрамлен `---`.

5. **Схема YAML обязана содержать:**
   - `domain`: [string] Один из: gaming, infra, ai, architecture, memes, research, megasoftware и т.д. (или из System Override).
   - `category`: [string] Узкая подкатегория.
   - `title`: [string] Емкий заголовок (EN).
   - `title_ru`: [string] Емкий заголовок (RU).
   - `summary`: [string] Выжимка сути на 2-3 предложения. (L1 память).
   - `tags_en`: [array] Список ключевых слов (EN).
   - `tags_ru`: [array] Список ключевых слов (RU).
   - `source`: [string] Источник данных (talkman, user, web).

**ПРИМЕР ВЫВОДА:**
---
domain: "infra"
category: "clickhouse"
title: "L0/L1 Knowledge Base Schema"
title_ru: "Схема L0/L1 базы знаний"
summary: "DDL скрипт для создания таблицы kb.articles с использованием ReplacingMergeTree и Bloom фильтров."
tags_en: ["database", "schema", "architecture", "bloom_filter"]
tags_ru: ["база_данных", "схема", "архитектура", "блум_фильтр"]
source: "user"
